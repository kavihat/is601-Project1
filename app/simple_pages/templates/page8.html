The Calculator Program

In this page, we will be using the example of a Calculator program to explain the SOLID programming principles. We will also be highlighting some design patterns used in this program.

To begin with, an overview of the Calculator program. The main interface is the Calculator class. A user can perform simple arithmetic operations using the appropriate methods in this class.

class Calculator:
    @staticmethod
    def add(value1, value2):
        addition = Add(value1, value2)
        return addition.get_result()

    @staticmethod
    def subtract(value1, value2):
        ...

    @staticmethod
    def multiply(value1, value2):
        ...

    @staticmethod
    def divide(value1, value2):
        ...

Each method in the Calculator class creates objects of the Operation type, which is an abstract base class for the Add, Subtract, Multiply and Divide classes, which perform the actual arithmetic operations, given two numbers. 

class Operation(ABC):
    @abstractmethod
    def get_result(self):
        pass

class Add(Operation):
    def __init__(self, value1, value2):
        self.__value1 = value1
        self.__value2 = value2

    def get_result(self):
        return self.__value1 + self.__value2

class Subtract(Operation):
    ...

Using this design, we will now look at two popular concepts in Software Engineering and OOP: the SOLID principles, and Design Patterns.

SOLID

The SOLID programming principles was originated by Robert C. Martin, and it stands for five different conventions of programming. These principles help to develop software that can be maintained and extended throughout its lifetime. The SOLID principles do this by avoiding code smells, refactoring code and applying Agile software development.

If you follow these principles you can improve the readability of your code by working on its structure and its logical consistency.

  1. Single Responsibility Principle
  2. Open/Closed Principle
  3. Liskov Substitution Principle
  4. Interface Segregation Principle
  5. Dependency Inversion Principle

Single Responsibility Principle

The single-responsibility principle (SRP) is a computer-programming principle that states that every module, class or function in a computer program should have responsibility over a single part of that program's functionality, and it should encapsulate that part. 

def add(value1, value2):
    """Testing the add"""
    obj_a = Add(value1, value2)
    return obj_a.get_result()

class Add:
    def get_result(self):
        return self.__value1 + self.__value2

In our example, the Calculator.add() method has the responsibility of returning the sum, but not actually performing the addition. The Add.get_result() method has the single responsibility of performing an addition and returning its result.

Open Close Principle

According to this principle, software entities should be open for extension but closed for modification. Our Add, Subtract, Multiply and Divide classes are each responsible for one behavior, and are whole and complete in terms of functionality. 

class Add(Operation):
    """performing the addition"""
    def get_result(self):
        return self.__value1 + self.__value2

class Subtract(Operation):
    """performing the subtraction"""
    def get_result(self):
        return self.__value1 - self.__value2

In order to support any more operations, say logarithms, none of the existing classes need to be changed. Instead, the Operation interface can be further extended to a new Logarithm sub-class, which will handle the logarithm operation.

Liskov Substitution Principle

This principle states that we can substitute a subclass for its base class without affecting behavior. It is supported by the object-oriented concepts of inheritance and polymorphism. This basically means that objects of a superclass shall be replaceable with objects of its subclasses without breaking the application.

Consider the following testing code: 

def test_type_bad():
    a = Add(1, 2)
    assert(type(a) == Add)
    assert(type(a) == Add)

def test_type_good():
    a = Add(1, 2)
    assert isinstance(a, Add)
    assert isinstance(a, Operation)

The second assert in test_type_bad() will fail because type() does not obey Liskov Substitution. However, both the asserts will pass in test_type_good because isinstance() permits replacing a child class with its parent class.

Interface Segregation Principle

This principle states that a client should not be exposed to methods it does not need. It is closely related to the Liskov Substitution principle and Single Responsibility principle.

The Interface Segregation Principle is helpful because:

  • Client developers exposed to methods they don't need
  • Maintenance becomes easier
  • It follows the Single responsibility principle making it readable and structured

In our example, any modifications to the behavior of the Divide operation does not need to deal with the behavior of the other classes. The fact that there is no reference to the other classes (Single Responsibility Principle at work!) makes this possible.

Dependency Inversion Principle

This principle states that high level modules which provide complex logic, should be easily reusable and unaffected by changes made to low level modules. The high level module must depend on abstractions of lower level modules, and not on concrete implementations.

Our Calculator class can confidently call the get_result() method of any operation object it has created, because it only needs to depend on the behavior guaranteed by the Operation interface and not on the specific implementation of any of its sub-classes.


Design Patterns used in Calculator code

Next we will look at some Design Patterns that our Calculator program uses. Design Patterns are general solutions to common programming problems, that can be applied (thoughtfully) to a given situation. Consider the following bit of code from our Calculator example: 

class Calculator:
   @staticmethod
   def add(value1, value2):
       """Testing the add"""
       obj_a = Add(value1, value2)
       return obj_a.get_result()

. . .

class Add(Operation):
   def __init__(self, value1, value2):
       self.__value1 = value1
       self.__value2 = value2

   def get_result(self):
       return self.__value1 + self.__value2

This bit of code uses two design patterns.

Command Pattern

The Calculator class builds the request to be sent to the Add class from the arguments it is given, and uses the get_result() method to compile a response for this request object. This is the Command design pattern.

Command is a behavioral design pattern that turns a request into a stand-alone object that contains all the information about the request.

Facade Pattern

In our example, the Calculator acts as a gateway to the Add class. The user does not need to know how addition is implemented, or indeed how to build an Add object. Instead, the user only needs to interact with the add() method in the Calculator class. This is the Facade design pattern.

Facade is a structural design pattern that provides a simplified interface to a library, a framework, or any other complex set of classes. It comes under structural design pattern as this pattern adds an interface to existing system to hide its complexities.
