Git is a free and open source distributed version control system designed to handle everything from small to very large projects with speed and efficiency. Its goals include speed, data integrity, and support for distributed, non-linear workflows.

History

Git development began in April 2005 under Linus Torvalds, for development of the Linux kernel, with other kernel developers contributing to its initial development. He wanted a distributed source-control management (SCM) system, but none of the available free systems met his needs. Torvalds cited an example of an SCM system needing 30 seconds to apply a patch and update all associated metadata, and noted that this would not scale to the needs of Linux kernel development, where synchronizing with fellow maintainers could require 250 such actions at once

Why Git?

Git helps software developing teams manage changes to their source code over time. In other words, version control keeps track of every change in your code, and allows you to go back in time when something goes wrong. It is also really helpful to prevent concurrent work from conflicting when multiple people are working on the same project. An individual may be working on the sidebar navigation feature for a webpage, while another may be simultaneously updating the header. Version control systems facilitate the work of multiple individuals by allowing them to use different branches as part of one “file tree” and merge their updated code to one source of truth when it's ready.

Git Design

The design of Git is akin to that of a filesystem, with added content addressing and versioning. Git has two data structures — a mutable index (also called stage or cache) that caches information about the working directory and the next revision to be committed; and an immutable, append-only object database. The index serves as a connection point between the object database and the working tree.

Git is a distributed version control system. As with most other distributed version control systems, and unlike most client–server systems, every Git directory on every computer is a full-fledged repository with complete history and full version-tracking abilities, independent of network access or a central server. Git gives each developer a local copy of the full development history, and changes are copied from one such repository to another. These changes are imported as added development branches and can be merged in the same way as a locally developed branch.

Common Operations in Git

  • Clone: Downloads a copy of the project (remote repository) to your local computer.
  • Branch: Lists all your project branches and highlights your current branch.
  • Checkout: Selects the branch you want to work on
  • Pull: Pulls recent changes from the remote repository to your local files.
  • Add: Stages file that you would like to commit.
  • Commit: Saves the group of changes you staged to your local repository.
  • Status: Lists all the files that you have worked on and have not been saved in a commit yet.
  • Stash: Saves any changes you have made locally and it reverts the working directory to match the HEAD of your last commit.
  • Push: Saves your local committed changes into the remote repository so everybody else has access to your work.
  • Tag: Mark specific points in a repository’s history as being important, including designating release points.

Concepts in Git

  1. Repositories: A repository is a working folder under version control. Repositories in git contain a collection of files of various different versions of a project. Git stores the file versions in a hidden .git folder alongside other information it needs to manage code. These files are imported from the repository into the local server of the user for further updates and modifications in the content of the file, an operation called cloning. 
    git init creates a new repo from an existing local folder
git clone copies the contents of an existing repo to a local folder
  2. Branch: Branches are a divergence from the main line of the development of a project, and allows a developer to continue to do work without messing with that main line. Branches enable developers to code in isolation from other changes in development. Once everything's working, the branch and its changes are shared with the team. Others can experiment with the code in their own copy of the branch without it affecting the work in progress in their own branches. A new branch can be created one from any existing one, with a specific commit or tag as the starting point. Being lightweight and local to the development machine, branches are an integral part of the Git development workflow.
    git checkout < branch > switches to a different branch
git add and git commit to stage local changes and commit to the feature branch
git push to push committed changes to the remote feature branch
  3. Commit:

    A commit in Git is used to record the changes in the repository, denoting a save point in the project development history. Each commit contains the index data, with all local changes included in the commit, and the commit message, which is a description of the changes included. Commits are the snapshots of the project. Every commit is recorded in the master branch of the repository. We can recall the commits or revert it to the older version. Two different commits will never overwrite because each commit has its own commit ID. This ID is a cryptographic number created by a SHA (Secure Hash Algorithm) algorithm.

    In order to create a commit, all included stages need to be staged using a git add command. Then,

     git commit -m “Commit Message”
    performs the commit, adding a new commit ID to the local history of the repository.

  4. Merge:

    Merging is a procedure used to connect forked histories. It allows you to bring all of the new commits from a feature branch into the mainline branch or another development branch. Git merge will associate a series of commits into one unified history. Often, the result of a merge will be a separate new commit, the so-called "merge commit".

    git merge < branch >
    merges the specified branch to the currently active branch
  5. Tag:

    Tags are used to give meaningful names to certain points in the history of a Git repository. Tags are used to mark a commit stage as relevant. We can tag a commit for future reference. Typically, the tag functionality is used to mark release points.

    Git supports two types of tags: lightweight and annotated. A lightweight tag is simply a pointer to a specific commit, commonly used as a temporary tag. Annotated tags, however, are stored as full objects in the Git database and contain the tagger name, email, and date; have a tagging message. Annotated tags are more commonly used, so as to include all this information. 

    git tag -a < tag> -m “Description”
    creates a new tag at the current point in the branch history.
